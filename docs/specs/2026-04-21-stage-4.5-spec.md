## 1. Goal

> **Eliminate all remaining `primary.sqlite` access from outside the server
> process, so P1 (loose coupling) is fully enforced.**

Stages 1-4 closed the three loud callers (`backup.sh`, `maintenance.sh`
integrity, `watchdog.sh` WAL checkpoint). The [caller audit](../analysis/2026-04-20-integrity-caller-audit.md)
found **20 distinct call sites across 9 files** plus **4 unexpected coupling
classes** (UC1-UC5). Stage 4.5 is the cleanup pass that retires every
remaining direct-to-SQLite caller and wires them through HTTP.

Headline violators (ranked by risk from §Recommendations of the audit):

| Rank | Caller | Class | Frequency | Audit § |
|---|---|---|---|---|
| 1 | `vcontext-hooks.js:2316` `sqliteExec(..., 'VACUUM;')` | B | hourly (when gc gate trips) | UC1 / #11 |

## 2. Phase 1 — Requirements

### 2.1 Acceptance criteria

Each AC is testable post-implementation (see §6 Verification plan).

- **AC-1 — zero external sqlite3 against primary (JS).**
  `rg "spawnSync\\(['\"]sqlite3['\"]" scripts/vcontext-*.js` returns matches
  only at `scripts/vcontext-server.js:346` (emergency `.recover` during
  startup quick_check failure — Class B, kept deliberately per audit §3).
  Every other spawn is gone.

- **AC-2 — zero external sqlite3 against primary (shell).**
  `rg "sqlite3 \\\"?\\$?\\{?(PRIMARY|RAM_DB|VCTX|DB_RAM|DB_PATH)" scripts/`
  returns zero matches across `.sh` files. Emergency one-shots (e.g.
  `pre-outage.sh`) either POST to admin endpoints or print a one-liner
  `curl` command the operator can run manually.

- **AC-3 — `buildRouteTable` stops fork/exec per hook entry.**
  After the change, `buildRouteTable` either (a) reuses a readonly
  `better-sqlite3` singleton held by `vcontext-hooks.js` at module load,
  OR (b) calls `GET /admin/route-table`. Under load (100 tool-use hooks
  in 60 s) `ps axo comm` shows **zero** transient `sqlite3` CLI processes
  attributable to hooks. Option (b) is preferred per spec §P1 — see
  Design §3.2 trade-off.

- **AC-4 — `/admin/verify-snapshot` no longer stalls event loop 20 s.**
  Issuing `POST /admin/verify-snapshot` while concurrently curl-polling
  `/health` every 250 ms shows `/health` p99 < 50 ms throughout the
  verify run. Achieved by moving the per-snapshot integrity_check loop
  into a worker thread or chunking with `setImmediate` yields.

- **AC-5 — dead-path booby traps removed or guarded.**
  Every `/Volumes/VContext/vcontext.db` reference in `scripts/` either
  (a) is replaced by `$PRIMARY` + routed through a server endpoint, or
  (b) has an explicit `DEAD_PATH_GUARD` comment and an `exit 0` short-
  circuit with a warning to stderr. `rg "Volumes/VContext/vcontext.db"
  scripts/` returns zero unconditional calls.

- **AC-6 — `/health` stays <10 ms p99 under concurrent admin load.**
  Baseline: `ab -n 1000 -c 4 http://127.0.0.1:3150/health` gives p99 < 10 ms.
  While a caller runs `POST /admin/snapshot` + `POST /admin/vacuum` + `POST
  /admin/auto-tune` in parallel, `/health` p99 stays < 10 ms. (Server
  internally rate-limits and queues heavy ops; cheap endpoints bypass the
  queue.)

- **AC-7 — every new endpoint has bounded error surface.**
  Each new endpoint handles at least these error cases with **non-cascading
  errors** (i.e., the failure returns a JSON envelope, does NOT
  `process.exit` or corrupt server state):
  - missing backup file → `503 Service Unavailable` + `{ error:
    "backup_unavailable" }`
  - ramDb handle unavailable (e.g., mid-migration) → `503` +
    `{ error: "ramdb_unavailable", retry_after_ms }`
  - malformed request body → `400 Bad Request` + `{ error: "bad_input",
    details }`
  - rate-limit exceeded → `429 Too Many Requests` + `Retry-After` header

- **AC-8 — OpenAPI + ENDPOINTS_LIST in sync.**
  `docs/schemas/vcontext-api-v1.yaml` describes every new endpoint
  introduced in Stage 4.5. `scripts/vcontext-server.js` `ENDPOINTS_LIST`
  (around line 6419 per schemas/README.md) advertises every new path.
  CI / grep check confirms no drift.

## 3. Phase 2 — Design

### 3.1 Data model

**No schema changes.** Stage 4.5 is strictly a caller-redirect exercise.
Every new endpoint reads/writes existing tables via the server's
singleton `ramDb` / `ssdDb` handles. Specifically:

- `entries`, `api_metrics`, `skill_registry`, `route_table`,
  `audit_log` — all existing.
- The `backup` and `snapshot` endpoints use `ramDb.backup(tmpPath)`
  (better-sqlite3 page-incremental) identical to the current `doBackup`.

### 3.2 API / new admin endpoints

Seven new endpoints (or endpoint variants). Every one requires:

- Auth: `X-Vcontext-Admin: yes` header + bearer token from
  `~/skills/data/.vcontext-admin-token` (same gate as Stage 2-4).
- Rate-limit tracker: per-endpoint in-memory last-run timestamp +
  inflight-count. `429 Too Many Requests` with `Retry-After: <sec>`
  header when exceeded.
- Event emission: every successful call writes an `entries` row with
  `type='admin-op'` and `{ endpoint, duration_ms, caller_ua, result }`
  (parity with Stage 2-4 endpoints, per parent spec AC).

```
┌────────────────────────────────────────────────────────────────────┐
│ New endpoints (all under /admin, all JSON, all admin-gated)        │
├────────────────────────────────────────────────────────────────────┤
│ POST /admin/snapshot         ← replaces hooks.js cmdSnapshot       │
│ GET  /admin/route-table      ← replaces hooks.js buildRouteTable   │
│ GET  /admin/metrics/window   ← replaces cmdMetrics + maintenance.sh│
│ POST /admin/auto-tune        ← replaces maintenance.sh:137-149     │
│ POST /admin/vacuum           ← replaces hooks.js:2316 + maint:156  │
│ POST /admin/gc/dry-run       ← replaces hooks.js cmdGc scans       │
│ POST /admin/policy-check     ← replaces hooks.js cmdPolicyCheck    │
└────────────────────────────────────────────────────────────────────┘
```

#### 3.2.1 `POST /admin/snapshot`

**Replaces**: `vcontext-hooks.js:2408 cmdSnapshot → spawnSync('sqlite3',
[VCTX_RAM_DB, '.backup', dst])` (audit UC2 / #13).

Request:
```json
{ "label": "daily" | "weekly" | "pre-outage" | <custom-string>,
  "include_wal": false }
```

Response:
```json
{ "status": "ok" | "fail" | "skipped",
  "path": "/abs/path/to/snapshot.db",
  "size_bytes": 6482317312,
  "elapsed_ms": 42000,
  "ran_at": "2026-04-21T03:00:00Z" }
```

Rate-limit: 1 per 10 minutes per label (daily snapshot is once/day; test
flows may need manual override via `?force=1`).

Internal: reuses the `doBackup` helper but writes to `BACKUP_DIR/snapshots/
<label>-<yyyyMMdd-HHmm>.db` instead of overwriting `vcontext-backup.sqlite`.
Atomic rename; cleans up `.tmp-wal`.

Error cases (per AC-7): `503 ramdb_unavailable`, `503 snapshot_dir_full`,
`429` if called more than once per 10 min.

Caller replaces: `hooks.js cmdSnapshot()` becomes a pure HTTP wrapper.
`maintenance.sh:111` line `$NODE $HOOK snapshot daily` becomes
`curl -sS -X POST -H ... http://127.0.0.1:3150/admin/snapshot -d
'{"label":"daily"}'`.

#### 3.2.2 `GET /admin/route-table`

**Replaces**: `vcontext-hooks.js:461/485 buildRouteTable →
spawnSync('sqlite3', [VCTX_RAM_DB, 'SELECT ...'])` (audit #10, highest
frequency caller).

Request: query-string only.
```
GET /admin/route-table?include_deprecated=0&max_age_min=5
```

Response:
```json
{ "version": "2026-04-21T00:00:00Z",
  "rows": [
    { "skill_id": "investigate", "priority": "P2",
      "trigger_keywords": [...], "last_used_at": "..." },
    ...
  ],
  "cache_hit": true,
  "generated_at": "..." }
```

Rate-limit: none (cheap read, server-cached). Server maintains a TTL cache
(default 60 s) — 100 concurrent hooks share one cache pull.

Internal: single `ramDb.prepare(...).all()` call, cached result held in
memory for `max_age_min` minutes. Cache invalidates on `POST
/admin/skill-registry/*` mutations (existing endpoints).

Caller replaces: `buildRouteTable` becomes a ~10-line `fetch` (or
`http.get`) call with JSON parsing. Retains a local 60 s cache on the
hooks side to amortize the network hop.

**H3 note**: this is the single highest-frequency hook-path change.
Parent session should confirm "HTTP + local-cache" is preferred over
"hooks-side readonly better-sqlite3 singleton". Trade-off:

| Option | Pros | Cons |
|---|---|---|
| A — readonly singleton in hooks.js | Zero new endpoint surface, no net hop | Still a second connection on primary (CLASS C), violates P1 spirit |
| B — `GET /admin/route-table` (recommended) | True P1 loose coupling, server owns cache | One extra network call per hook entry (~2 ms loopback) |

Recommendation: **Option B**. Loopback 2 ms is negligible vs. the 20-50 ms
of `spawnSync` fork/exec today. Option A leaves a latent CLASS C connection
that will trip future audits.

#### 3.2.5 `POST /admin/vacuum`

**Replaces**: `vcontext-hooks.js:2316` hourly VACUUM (audit UC1 / #11,
**CRITICAL** per recommendations) + `vcontext-maintenance.sh:156` weekly
VACUUM (audit #15).

Request:
```json
{ "target": "primary" | "backup", "force": false }
```

Response:
```json
{ "status": "ok" | "skipped",
  "reason": "rate_limited" | "recent_vacuum" | null,
  "freed_bytes": 123456789,
  "elapsed_ms": 45000 }
```

Rate-limit: **1 per 7 days per target**, hard-enforced. Server persists
`last_vacuum_at` in `entries` with `type='admin-vacuum-run'`. Requests
inside the window return `skipped` unless `force=true` AND caller is
admin-plus (flag to be added — see Open questions §7).

Internal: `ramDb.exec('VACUUM;')`. Server self-pauses write-heavy
endpoints during the call (enqueues them, 429s incoming writes for the
vacuum window).

Caller replaces: `hooks.js:2316` deleted entirely (audit says "delete,
server auto-vacuum handles it"). `maintenance.sh:156` becomes a weekly
`curl POST /admin/vacuum -d '{"force":true}'` gated on `[ "$(date +%u)"
= 7 ]`.

### 3.3 Sequence (end-to-end, maintenance.sh cycle)

Current (pre Stage 4.5):
```
maintenance.sh (hourly)
  │
  ├─ $NODE hooks.js metrics  ── fork ── sqlite3 primary.sqlite SELECT ... × 6
  ├─ $NODE hooks.js policy   ── fork ── sqlite3 primary.sqlite SELECT ... × 6
  ├─ $NODE hooks.js gc       ── fork ── sqlite3 primary.sqlite SELECT ... + VACUUM  ← CRITICAL
  ├─ $NODE hooks.js snapshot ── fork ── sqlite3 primary.sqlite .backup    (daily, ~90 s)
  ├─ sqlite3 DB_RAM "..." × 8 (dead path, would re-awaken on RAM remount)
  └─ server.js keeps running — BUT WAL pinned by snapshots above
```

After Stage 4.5:
```
maintenance.sh (hourly)
  │
  ├─ curl GET  /admin/metrics/window?hours=1&metric=all    → 1 HTTP call
  ├─ curl POST /admin/policy-check  -d '{"checks":[...]}'   → 1 HTTP call
  ├─ curl POST /admin/gc/dry-run     -d '{}'                → 1 HTTP call
  ├─ curl POST /admin/gc/apply       -d '{...}'             → existing endpoint
  ├─ curl POST /admin/snapshot       -d '{"label":"daily"}' → 1 HTTP call (gated on schedule marker)
  ├─ curl POST /admin/auto-tune      -d '{}'                → 1 HTTP call
  └─ (weekly only) curl POST /admin/vacuum -d '{"force":true}'
                                                  ↓
                                server.js's ramDb — sole file opener
```

Zero forks, zero SQLite file access, zero read-snapshot pinning on primary.

Sequence for `buildRouteTable` hot path:
```
hook entry (UserPromptSubmit / PostToolUse)
  │
  └─ node vcontext-hooks.js routeSkills
      │
      ├─ local cache still fresh (<60 s)? ── yes ──> return cached rows
      └─ no ──> GET /admin/route-table?max_age_min=1
              │  (~2 ms loopback)
              └─ server returns cached rows (server-side TTL also 60 s)
                  → hooks.js refreshes local cache → return rows
```

Cache layering means a burst of 100 hook entries inside 60 s produces
**one** HTTP call to the server.

### 3.4 Constraints

- **Must not break existing clients.** All additions are new routes; no
  removal of existing `/admin/*` paths. `hooks.js` keeps its CLI surface
  (`node vcontext-hooks.js <subcmd>`) for manual debug — subcommands
  internally call HTTP instead of sqlite3, so CLI still works when
  server is up. CLI gracefully degrades with `"server offline; this
  subcommand is unavailable"` when `/health` fails (fail-open for debug).
- **OpenAPI (`docs/schemas/vcontext-api-v1.yaml`) must stay in sync.**
  Each new endpoint adds a `paths:` entry same-commit as the server
  code. CI check (if any) enforces no drift.
- **ENDPOINTS_LIST in `vcontext-server.js` (~L6419) lists every new
  path.** Same commit as server code.
- **No admin token handoff changes.** Reuse existing
  `~/skills/data/.vcontext-admin-token` pattern; no env-var
  proliferation.
- **Backpressure stays local.** Rate-limit state is per-process, in-
  memory; no Redis/distributed-lock dependency added.
- **Loopback binding only.** New endpoints rely on 127.0.0.1 binding and
  existing loopback-trust. No remote exposure (per security review
  default).

## 5. Risk register

| # | Risk | Probability | Impact | Mitigation |
|---|---|---|---|---|
| R1 | `buildRouteTable` HTTP cutover causes latency regression in hot hook path | MED | HIGH (user-visible slowness on every tool use) | Keep 60 s local cache in hooks.js; preserve `--fallback-to-sqlite` flag (gated by env var) for 24 h; canary via SessionStart hook metric comparing hook-to-first-tool time pre/post |

## 7. Rollback plan

Every commit is a single-file or two-file change with a corresponding
LaunchAgent NONE (no plist touched in Stage 4.5) OR, in the dead-path
cleanup cases, a plist-adjacent shell script only — revertable by `git
revert <sha>` without service restart.

| Commit | Rollback trigger | Revert action |
|---|---|---|
| 1 | Gc stops working / migration regressions | `git revert <sha>` — VACUUM re-enabled; takes max 1 hourly cycle to recover |
| 2 | `/admin/vacuum` accidentally trips in prod | Flip env `VCTX_VACUUM_DISABLED=1` (guard in handler); revert if env insufficient |
| 3 | Snapshot or auto-tune breaks backup cycle | Revert; maintenance.sh still uses old `$NODE $HOOK snapshot` until Commit 7 |
| 4 | Metrics / policy-check / gc endpoints error | Revert; hooks.js still uses sqlite3 spawn until Commit 7 |
| 5 | Worker/chunking introduces verify inconsistencies | Revert; verify-snapshot reverts to inline loop — 20 s stall returns but correctness OK |
| 6 | Maintenance cycle fails | Revert single file; old sqlite3-spawn maintenance.sh returns |
| 7 | hooks.js HTTP swap broken | Revert; sqlite3 spawns return until fix |
| 8 | pre-outage.sh or self-improve breaks | Revert single file per script |
| 9 | **CRITICAL** — routing regressions break hook infra | (1) env var `VCTX_ROUTE_TABLE_MODE=sqlite` re-enables fallback without revert; (2) if (1) insufficient, `git revert <sha>` — pre-cutover hooks.js returns |
| 10 | Guard asserts false-positive | Revert asserts; keep doc updates |

No plist changes → no LaunchAgent rollback. If a plist needs adjustment
(e.g., to add a maintenance cron flag in Commit 6), follow same revert
pattern + `launchctl unload/load` cycle for affected plist.
