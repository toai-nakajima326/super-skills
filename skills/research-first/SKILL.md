## Rules

1. **Never say "impossible", "not available", "doesn't exist", or "no way to do this" without first researching**
   - Search the web for the latest documentation
   - Check changelogs, GitHub issues, and release notes
   - Look for workarounds, alternative approaches, or recent additions

2. **Assume your knowledge is stale**
   - Your training data has a cutoff date
   - APIs, tools, and platforms add features frequently
   - What was impossible 3 months ago may now be possible

3. **Research before responding**
   - When a user asks "can X do Y?" — search first, answer second
   - When you think a feature doesn't exist — verify before stating it
   - When you're about to say "unfortunately" — stop and research

4. **If research confirms it's truly impossible, explain why with evidence**
   - Link to the relevant documentation
   - Explain what the closest alternative is
   - Suggest workarounds

## Workflow

1. User asks about a capability
2. **STOP** — do not answer from memory alone
3. Search for the latest information (web search, docs, changelogs)
4. Answer based on current facts, not training data assumptions
5. If the capability exists but you didn't know, implement it
6. If it truly doesn't exist, explain with evidence and suggest alternatives

## Gotchas

- Saying "this is not possible" when it actually is destroys user trust
- The cost of a quick search is far lower than the cost of a wrong "no"
- This applies to ALL tools and platforms, not just Claude Code
- Even if you're 90% sure something doesn't exist, the 10% matters
